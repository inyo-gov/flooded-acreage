# Flooded Acreage Monitoring Dashboard

This project provides automated monitoring of flooded acreage in the Blackrock Waterfowl Management Area using **Sentinel-2 optical** imagery. The system generates individual flood reports and a dashboard with monthly time series visualization. **Sentinel-2 is the operational method** for monthly acreage and the public site; results have been reasonable for open water and many managed units. **Sentinel-1 SAR** scripts exist in the repo but are **not yet incorporated** into the dashboard workflow—planned for areas where **emergent vegetation** (e.g. cattails) masks the water signal in optical data.

Currently the requirement to quantify flooded acreage has been implemented by field workers physically walking the wetted extent with a handheld GPS, which can take up to 8 person days in the field every time it is needed. The current requirements mandate this be done around November 1 and March 1 time periods. This application provides: 1) instantaneous flooded acreage estimates for the Nov 1 and Mar 1 periods, rather than multiple days of field work, and 2) acreage estimates throughout the year, every five days or whenever cloud-free Sentinel scenes are available. 

Sentinel 2 provides 10m resolution in Band 8 near-infrared, whereas Landsat resolution is 30m. The improved resolution of Sentinel over Landsat and improvements in availability of this imagery from earth engine's archive drove the development of this work. There are thousands of papers, and dozens of water indices used to quantify open water, yet they all use some combination of near infrared, short-wave infrared, or ratios of these. Here a simple thresholding of NIR is where I've started and it yields good results with a few validation points that will be obtained for the official Nov 1 and Mar 1 time periods to help calibrate the threshold for reporting purposes.

## Introduction

This project computes flooded acreage within BWMA units from satellite data and publishes results through the Quarto dashboard.

**In production: Sentinel-2 (optical)**

- Monthly reports and the dashboard use `flood_report.py` only.
- Cloud-masked **median composite** over a 15-day window from the 1st of the month; NIR threshold for open water.
- Works well for **open water** and many flooded units; field validation supports threshold calibration.

**Upcoming: Sentinel-1 (SAR) and fusion**

- Scripts `flood_report_s1.py`, `flood_report_fusion.py`, and `sar_utils.py` are **experimental**—built and tested, but **not** part of the official monthly workflow or GitHub Pages dashboard yet.
- **Why add SAR:** Where **emergent vegetation overtops or obscures** the water surface, optical NIR can miss wet acreage. C-band SAR change detection (wet vs dry baseline) is intended to detect flooding **under** that vegetation.
- **Fusion** (union / intersection / confidence classes) would combine S2 open-water strength with S1 under-canopy sensitivity once SAR is production-ready.

The analysis calculates flooded area within predefined units and generates HTML reports and CSV exports.

### Installation

1. Clone the repository:

   ```sh
   git clone https://github.com/inyo-gov/flooded-acreage.git
   cd flooded-acreage
   ```

2. Create and activate a virtual environment:

   ```sh
   python -m venv venv
   source venv/bin/activate  # On Windows use `venv\Scripts\activate`
   ```

3. Install the required libraries:

   ```sh
   pip install -r requirements.txt
   ```

4. Authenticate with Google Earth Engine:

   ```sh
   earthengine authenticate
   ```

5. (Optional) Set Earth Engine Project ID:

   If you want to use a different Google Cloud project, set the environment variable:
   
   ```sh
   export EARTH_ENGINE_PROJECT_ID=your-project-id
   ```
   
   If not set, scripts will use the default project ID. Note: Project IDs are identifiers, not secrets, but you can keep them private by using this environment variable.

## Usage

### Generate Sentinel-2 Flood Report

Use the Python script to generate a flood report from Sentinel-2 optical imagery:

```sh
python flood_report.py <date> <threshold>
```

Example (monthly BWMA summary — use the **1st of the month** as the start date):

```sh
python flood_report.py '2026-06-01' 0.14
```

#### Monthly BWMA acreage workflow (for staff)

This repo’s **dashboard** (`index.qmd` → `docs/index.html`) is updated from Sentinel-2 reports generated by `flood_report.py`. The steps below are the intended monthly handoff once Earth Engine access is set up.

**When to run**

- Run **around the middle of the month** (e.g. the 12th–16th), not on the 1st.
- Use a **start date of the 1st** of that month (`YYYY-MM-01`). The script searches a **15-day window** (1st through the 16th).
- By mid-month there are usually **several** Sentinel-2 passes in the archive (often on the order of ~5–7 scenes). Waiting avoids a report based on only one early-month image.

**What the script does (composite, not a single scene)**

- Collects all Sentinel-2 surface reflectance images in the window (cloud cover &lt; 50%, clipped to the BWMA box).
- Masks clouds per scene (Sentinel **SCL**), then builds a **median composite** across those scenes.
- Flooded acreage uses **NIR (B8)** on that composite: pixels below the reflectance threshold count as water.
- This is **more robust** than thresholding one acquisition: a cloudy or odd pass is less likely to dominate the month.

**Report filenames vs. imagery**

- Console output includes `Number of images in collection:` — that is how many scenes fed the median.
- The HTML/CSV names use an **“Image date”** taken from the **first** image in the collection (often the 1st of the month). That label is for naming only; the acreage still reflects the **composite** over the full window.

**Threshold**

- Current monthly reports use **`0.14`** NIR reflectance (see validation notes elsewhere in this README). Re-calibrate from field GPS if wet/dry checks drift.

**After `flood_report.py`**

1. Confirm outputs under `flood_reports/reports/` and `flood_reports/csv_output/` (repo root; gitignored).
2. Render the site: `quarto render` (or `quarto render index.qmd`).
3. In `index.qmd`, set the **Most Recent Report** iframe `src` to the new `flood_reports/reports/bwma_flood_report_<date>_<threshold>.html` if you want that month featured (the monthly table links are built automatically from the report files at render time).
4. Commit and push **`docs/`** (and source changes) so GitHub Pages updates. Published report files live under **`docs/flood_reports/`** after render.

**Requirements:** Python deps, `earthengine authenticate`, and Earth Engine project access (`EARTH_ENGINE_PROJECT_ID` optional; see Setup).

### Sentinel-1 SAR and fusion (upcoming — not in dashboard yet)

The following scripts are **future work**. Staff should use **`flood_report.py`** for official monthly acreage until SAR is integrated and validated.

**Planned role:** SAR will help where **emergent vegetation** covers the water and the optical NIR signature is suppressed; open-water units can continue to rely primarily on Sentinel-2.

#### Sentinel-1 SAR flood report (experimental)

```sh
python flood_report_s1.py <start_date> <dry_start> <dry_end> [--dvv-thresh] [--vv-vh-ratio] [--orbit-pass]
```

Example:
```sh
python flood_report_s1.py '2024-10-22' '2024-08-01' '2024-08-31' --dvv-thresh -1.5
```

**Parameters:**
- `start_date`: Start date for wet/flood period (YYYY-MM-DD)
- `dry_start`: Start date for dry baseline period (YYYY-MM-DD)
- `dry_end`: End date for dry baseline period (YYYY-MM-DD)
- `--dvv-thresh`: dVV threshold in dB (default: -1.5). Lower values = more sensitive
- `--vv-vh-ratio`: Maximum VV-VH ratio in dB (default: 3)
- `--orbit-pass`: Filter by orbit direction: ASCENDING or DESCENDING (optional)

#### Fused S1+S2 report (experimental)

```sh
python flood_report_fusion.py <start_date> <s2_threshold> <dry_start> <dry_end> [--s1-thresh] [--fusion-mode] [--orbit-pass]
```

Example:
```sh
python flood_report_fusion.py '2024-10-22' 0.22 '2024-08-01' '2024-08-31' --fusion-mode union
```

**Fusion modes (when adopted):**
- `union` (default): Flood if either S1 or S2 detects it
- `intersection`: Flood only if both S1 and S2 detect it (conservative)
- `confidence`: Create confidence classes (1=S2-only, 2=S1-only, 3=both)

**Parameters:**
- `start_date`: Start date for wet/flood period (YYYY-MM-DD)
- `s2_threshold`: S2 NIR threshold (same as flood_report.py)
- `dry_start`: Start date for S1 dry baseline period (YYYY-MM-DD)
- `dry_end`: End date for S1 dry baseline period (YYYY-MM-DD)
- `--s1-thresh`: S1 dVV threshold in dB (default: -1.5)
- `--vv-vh-ratio`: S1 VV-VH ratio threshold in dB (default: 3)
- `--fusion-mode`: Fusion mode: union, intersection, or confidence (default: union)
- `--orbit-pass`: S1 orbit pass direction: ASCENDING or DESCENDING (optional)

### Parameters (Sentinel-2 flood report)

- **Surface reflectance threshold**: NIR (B8) cutoff for water; applied to the **median composite**, not a single scene.
- **Start date**: First day of the search window (use **`YYYY-MM-01`** for monthly summaries).
- **End date**: Start date + 15 days (computed in the script).
- **Imagery**: All qualifying Sentinel-2 scenes in that window, cloud-masked, then **median** composite.
- **Bounding box**: Fixed BWMA extent in `flood_report.py`.

## Technical Notes

### Sentinel-2 (Optical) — production

- **Surface reflectance:** Sentinel-2 SR, 10 m NIR, ~5-day revisit; median composite over the monthly window (see staff workflow above).
- **Flood detection:** NIR thresholding identifies **open water** well; this is what the dashboard uses today.
- **Limitation:** Can **under-estimate** flooding where **emergent vegetation** (e.g. cattails) covers the surface and the water no longer looks like open water in NIR—that is the main motivation for future SAR work.

### Sentinel-1 (SAR) — upcoming

- **Not yet in the monthly dashboard.** Scripts exist for development and testing.
- **Intended use:** Detect wetness **under emergent vegetation** when optical NIR is masked; C-band SAR change detection (wet period vs dry baseline) responds to water beneath canopy structure.
- **Indicators:** dVV (VV backscatter change), VV/VH ratio; requires a **dry baseline** period (often summer).

### Fusion — upcoming

- When adopted, combine S2 (open water) with S1 (under-vegetation) via union, intersection, or confidence classes.
- **Goal:** Fuller extent where cattails or similar emergent veg overtop the water optical signature, without losing S2 strength on open units.

### General
- **Google Earth Engine API**: All analysis performed using the Earth Engine Python API for processing large geospatial datasets.
- **Dashboard Technology**: The dashboard is built with Quarto, providing interactive time series visualization and report management.
- **Data Management**: Historical data is automatically tracked and visualized in the dashboard.
- **Reports**: Individual HTML reports are generated for each analysis, with CSV data export capabilities.
- **Field Validation**: Results are validated during routine field checks throughout the seasonal flooding cycle (September through April). Field data collection in ArcGIS Online will inform future threshold calibration and relative elevation analysis needs.

## Model Validation

The flood mapping model is validated using field-collected GPS points from biologists during bird surveys.

### Field Data Collection

Biologists collect GPS points (yes/no flooded) using ArcGIS Online during regular bird surveys. See `docs/field_validation_protocol.md` for detailed collection guidelines.

### Running Validation Analysis

After exporting validation data from ArcGIS Online to GeoJSON or CSV:

```sh
python validate_flood_model.py <validation_data> <report_date> [--threshold]
```

Example:
```sh
python validate_flood_model.py data/validation/validation_points_2025-12-08.geojson 2025-12-08 --threshold 0.18
```

This generates:
- **Validation Report (HTML)**: Accuracy metrics, confusion matrix, spatial analysis
- **Validation Points CSV**: Point-by-point comparison of observed vs predicted
- **Validation Map (HTML)**: Interactive map showing correct/incorrect predictions

### Validation Metrics

The validation analysis calculates:
- Overall Accuracy
- Precision (of predicted flooded, how many are actually flooded?)
- Recall/Sensitivity (of actually flooded, how many did we detect?)
- Specificity (of actually dry, how many did we correctly identify?)
- F1 Score
- Accuracy by unit

Results help calibrate the NIR threshold and improve model performance over time.
